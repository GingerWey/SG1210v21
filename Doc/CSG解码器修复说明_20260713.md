# CSG解码器修复说明

**日期**: 2026-07-13  
**版本**: CSGDecoder V1.54  
**修复人**: Wey. Silver Grid

## 问题描述

### 原始问题
CSG流式解码器在 `/cas None` 模式下，使用indexed color（crn=4/8）时，出现**4列错位**现象：
- 表现：每行的最后4个像素出现在下一行的前4列
- 影响范围：所有使用 `/cas none` + indexed color 的图片
- 测试用例：
  - `LV_CrossMark16x16Green.png /crn 8 /cas none`
  - `MU_Home24x22.png /crn 4 /cas none`
  - `MU_Item32x32_01.png /crn 4 /cas none`

## 根本原因

### bitCount字段的双重语义
`CSGDecoderState::bitCount` 在不同压缩算法中有不同含义：

| 压缩算法 | bitCount含义 | 初始值应为 |
|---------|-------------|-----------|
| CAS=0 (None) | bit accumulator，bit-unpacking时的已填充位数 | **0** (空buffer) |
| CAS=1 (RLE) | bit accumulator | **0** (空buffer) |
| CAS=2 (DEFLATE) | 由Huffman管理，此处不使用 | **0** |
| CAS=3 (MiniLZ77) | token index (0-7) 或 8=需要读取新控制字节 | **8** (读新ctrl) |

### 错误的初始化代码
```cpp
// CSGDecoder.cpp:341 (修复前)
state->bitCount = 8;  // ❌ 对所有压缩算法都设为8
```

对于CAS=0的indexed color模式：
1. DecodeNone在bit-unpacking时，从bitCount=8开始
2. 第一次进入循环，认为buffer中已有8 bit数据
3. 实际bitBuf=0，这8 bit全是无效数据
4. 导致前8个index被错误解析，产生4列错位（8 index / 2 bpi = 4列）

## 修复方案

### 1. 条件初始化bitCount
```cpp
// CSGDecoder.cpp:364 (修复后)
state->bitCount = (static_cast<int>(CompressAlgorithm::kMiniLZ77) == state->cas) ? 8 : 0;
```

**逻辑**：
- 如果是MiniLZ77（CAS=3）→ bitCount=8（需要读取第一个ctrl字节）
- 其他算法（CAS=0/1/2）→ bitCount=0（bit buffer为空）

### 2. 替换magic number为枚举常量
提升代码可维护性，避免magic number：

#### 修改位置1：MiniLZ77 dispatch判断
```cpp
// CSGDecoder.cpp:873 (修复前)
if (3 == state->cas) {

// CSGDecoder.cpp:873 (修复后)
if (static_cast<int>(CompressAlgorithm::kMiniLZ77) == state->cas) {
```

#### 修改位置2：CAS switch分支
```cpp
// CSGDecoder.cpp:980-987 (修复前)
case 0:  // None
case 1:  // RLE
case 2:  // DEFLATE

// CSGDecoder.cpp:980-987 (修复后)
case static_cast<int>(CompressAlgorithm::kNone):
case static_cast<int>(CompressAlgorithm::kRLE):
case static_cast<int>(CompressAlgorithm::kDEFLATE):
```

## 技术细节

### DecodeNone bit-unpacking逻辑
```cpp
// CSGDecoder.cpp:489-509
while (dec < toDecode) {
    // Refill bitBuf when empty
    while (bitCount < bpi && ip < ipEnd) {
        bitBuf = (bitBuf << 8) | input[ip++];
        bitCount += 8;
    }
    
    if (bitCount >= bpi) {
        // Extract CRI index
        uint8_t cri = (bitBuf >> (bitCount - bpi)) & bpiMask;
        bitCount -= bpi;
        // Write to output with palette lookup
        WriteCrmPixel(output, dec, bpc, pal, cri);
        ++dec;
    } else {
        break;  // Not enough bits, need more data
    }
}
```

**关键点**：
- `bitCount`表示`bitBuf`中已填充的有效位数
- 初始必须为0，否则会误认为buffer中有数据
- 每次从input读取8 bit时，bitCount += 8
- 每次提取bpi个bit后，bitCount -= bpi

### MiniLZ77 tokenIdx语义
```cpp
// CSGDecoder.cpp:878-913
int tokenIdx = state->bitCount;  // 0-7: current token, 8: need new ctrl
while (dec < toDecode) {
    if (8 <= tokenIdx) {
        ctrl = input[ip++];  // Read new control byte
        tokenIdx = 0;
    }
    
    if (0 != (ctrl & (1u << tokenIdx))) {
        // Match token: read distance & length
    } else {
        // Literal token: read raw byte
    }
    
    ++tokenIdx;
}
```

**关键点**：
- `tokenIdx` = 当前token在ctrl字节中的位索引（0-7）
- `tokenIdx` = 8 是sentinel值，表示需要读取新的控制字节
- 初始必须为8，确保第一次循环就读取ctrl
